OVERVIEW:
---------
Site Connect can be complex to configure. What makes Site Conect work is enabling your web site
to make API calls against the Ringside server. This means your web site will need:

1) The SiteConnectClient class (included with the rest of the Ringside clients)
2) A site-specific API key and secret (included in the database configuration block, below); in practice, these will need to
   be generated by the operator of the Ringside server
3) A way to manage user logins and identities (currently, only integer user IDs are supported)
4) A social application. This demo uses the Footprints sample application.

WEB CONFIGURATION:
------------------

NOTE: The URLs used throughout this example point to the example.com domain. This domain
is reserved for examples (how fortunate for all of us!). In order to run the demo as-is, you
will need to update your host configuration (/etc/hosts or %WINDOWS%\system32\drivers\etc\hosts)
to match the following:

127.0.0.1       social.example.com www.example.com

The "social" sub-domain serves up the social applications (widgets), and the "www" subdomain is
the example site. You should access the sample site using www.example.com on the port on which your
server is installed (typically 80 or 8080). For example, your demo Site Connect site can probably be
reached at the following URL:

http://www.example.com:8080/demo/siteconnect/
